‚ĚĮ does MPI have to be installed in the base environment in order to use MUMPS   

  Searched for 1 pattern (ctrl+o to expand)

‚óŹ No. MPI just needs to be in the active project environment, not the
  base/global one.

  In this repo it already is ‚ÄĒ Project.toml lists MPI, MPIPreferences, and MUMPS
   as direct deps, and MUMPS_jll pulls in MPI binaries via JLL artifacts
  (MPICH_jll, OpenMPI_jll, etc.). So as long as you launch Julia with
  --project=. (or activate the EarthBox project), MUMPS works without anything
  installed in @v1.x.

  The only time you'd touch a global install is if you wanted a system MPI (e.g.
   on an HPC cluster) ‚ÄĒ and even then the configuration goes through
  MPIPreferences.use_system_binary() inside the project, written to
  LocalPreferences.toml, not to the base env.

‚ĚĮ Then how could an mpi test be desined that would run via CI on github         

  Read 1 file, listed 2 directories (ctrl+o to expand)

‚óŹ Short version: in CI you don't actually need a "base env" ‚ÄĒ that's a
  user-ergonomics step (so mpiexecjl is on $PATH for interactive use). CI can do
   everything inside the project env.

  Three building blocks:

  1. Pin MPI to the JLL before tests run. Add a step (or a test/setup_mpi.jl
  invoked in CI) that writes LocalPreferences.toml:
  using MPIPreferences
  MPIPreferences.use_jll_binary("OpenMPI_jll"; force=true)
  Has to run in a separate julia invocation before any process loads MPI, since
  prefs are read at MPI load time.

  2. For most MUMPS tests: single-rank, in-process. MUMPS works fine with comm
  size 1, so no launcher needed:
  using MPI, MUMPS, SparseArrays, Test
  MPI.Initialized() || MPI.Init()
  A = sparse([2.0 1; 1 3]); b = [3.0, 4]
  x = MUMPS.solve(A, b)
  @test A*x ‚Čą b
  Run this as its own Julia subprocess from runtests.jl so MPI.Init/Finalize
  doesn't contaminate the rest of the suite.

  3. For genuine multi-rank tests: launch via MPI.mpiexec() from Julia ‚ÄĒ no
  mpiexecjl on PATH required:
  using MPI
  mpiexec() do exe
      run(`$exe -n 2 $(Base.julia_cmd()) --project=.
  test/tests/mumps_mpi_worker.jl`)
  end
  The worker file does its own MPI.Init(), runs the MUMPS solve, asserts, exits.
   The parent test just checks the subprocess exit code.

  CI workflow change is minimal ‚ÄĒ one extra step between julia-buildpkg and
  julia-runtest:
  - name: Configure MPI to use OpenMPI_jll
    run: julia --project=. -e 'using MPIPreferences; 
  MPIPreferences.use_jll_binary("OpenMPI_jll"; force=true)'

  Then gate the MUMPS tests behind a new group (e.g. EARTHBOX_TEST_GROUP=mumps)
  so they only run when you want them.

  The README instructions remain correct for humans developing locally ‚ÄĒ they
  just don't apply to a fresh CI runner.
