‚ĚĮ Are there any negative downsides to doing GC.gc()                             

‚óŹ Yes, a few real ones ‚ÄĒ worth being honest about before adding them:

  1. Pause time. GC.gc() is a stop-the-world full collection. On a 50+ GB
  process it can take 100‚Äď500 ms per call (sometimes more ‚ÄĒ it has to trace the
  whole heap). Three calls per output step ‚Čą 0.3‚Äď1.5 s added to export time.
  Small relative to your 140 s export, but not free.

  GC.gc(false) does an incremental collection ‚ÄĒ much faster (~50‚Äď100 ms) but may
   not reclaim everything in one pass. Usually good enough for a single large
  recently-out-of-scope array.

  2. Memory may not be returned to the OS. GC reclaims memory at the Julia
  level, but Julia's allocator (jemalloc/malloc) may keep freed regions in its
  pool rather than returning them to the kernel. So live Julia memory drops, but
   RSS may not. For our case this is mostly fine ‚ÄĒ the large arrays come from
  big allocations that typically do get returned ‚ÄĒ but Sys.maxrss() being the
  peak, not current, means we won't see a drop either way. We'll know it worked
  because the next write's burst won't stack on top.

  3. Interferes with Julia's GC heuristics. Julia's GC is tuned to run under
  allocation pressure. Manual calls can throw off that tuning and occasionally
  cause more total GC time elsewhere. In a tight computation loop this can
  matter; in an I/O-bound output step it's negligible.

  4. Brittle signal of intent. A future maintainer sees GC.gc() in the middle of
   a function and wonders why. Mitigation: a one-line comment like # force 
  reclaim of IDs buffer before the X write to avoid stacking.

  5. Doesn't help if a reference leaks. GC only reclaims truly unreferenced
  memory. If JLD2 internally retains a reference to the ID array for deferred
  flush, GC won't touch it. This is the scenario where it'd simply have no
  effect ‚ÄĒ which is a "no harm, no gain" outcome, not a regression.

  None of these affect correctness. GC cannot change program semantics ‚ÄĒ only
  timing and memory shape.

  My take: start with GC.gc(false) (incremental) between IDs/X/Y. Small pause
  cost, no heuristics disruption. Add a single-line comment on each so the
  intent is obvious. If the probe confirms it works, we keep them. If not, no
  harm done ‚ÄĒ remove them and try Option B (disable compression).

  Want me to proceed with the incremental variant?

                            WITHOUT GC       WITH GC(false)      őĒ
  before IDs:               47,410 MB        47,411 MB           -
  before X:                 49,394 MB        49,395 MB           -
  before Y:                 51,222 MB        49,395 MB           -1,827 MB ‚úď
  before scalar loop:       51,222 MB        49,395 MB           -1,827 MB ‚úď
  scalar 5/11:              51,683 MB        51,475 MB           -208 MB
  final peak:               51,683 MB        51,475 MB           -208 MB

  The GC calls did work as intended ‚ÄĒ X no longer stacks on IDs residual, Y no
  longer stacks on X. But the net savings are only 208 MB because the scalar
  loop became the new bottleneck. Here's why:

  The scalar loop was hidden behind the X+Y stack before. Each scalar write is
  also ~2 GB burst (same pattern: ~900 MB array + ~900 MB compression buffer +
  JLD2 overhead). Previously, those bursts fit under the 51,222 MB peak from the
   stacked X/Y; now with a lower baseline (49,395 MB), the scalar bursts push
  the peak up to 51,475 MB anyway.

  This is the real insight: every compressed write in this file produces a ~2 GB
   burst, regardless of array size. The peak equals baseline + ~2 GB, no matter
  how many writes or how we order them. GC can prevent stacking between writes
  but can't shrink the single-write burst.

  Where that leaves us

  Given compression is a hard requirement, the ~2 GB per-write burst is
  structural ‚ÄĒ it's the JLD2/zlib output buffer. Reducing it further means
  attacking the burst size itself:

  1. Keep the change as-is. 208 MB is real. Probes are noise-free and ship-able.
  - Total improvement from pre-optimization: 56,238 ‚Üí 51,475 = 4,763 MB saved
  - Add a brief comment that the GC calls are intentional memory-shape
  management

  2. Chunked writes via JLD2's low-level API ‚ÄĒ could shrink the compression
  buffer to chunk size (e.g. 100 MB instead of 900 MB), potentially cutting the
  burst from ~2 GB to ~1 GB.
  - Requires digging into JLD2's Dataset / chunking API
  - Correctness: still bit-exact on read (HDF5 chunking is transparent to
  readers)
  - Risk: more invasive change, needs careful testing

  3. Use a smaller dtype for the IDs array. collect(Int32, 0:nmarkers-1) cuts
  the IDs array size from 910 MB to 455 MB. But the IDs burst isn't the dominant
   one anymore ‚ÄĒ the scalar loop is ‚ÄĒ so this wouldn't change the peak. Not
  worth doing in isolation.

  4. Stop here. We've gone from 56 GB ‚Üí 51.5 GB (~4.8 GB saved) across Phase 3 +
   GC. The remaining spike is structural, and further improvement is expensive
  in code complexity.

  My honest take: option 1 (keep it) or option 4 (stop). The GC calls are cheap,
   safe, and produce measurable (if modest) savings. Chunked writes would take
  real engineering effort and might not move the needle much more. Would you
  like to remove the probes and finalize, or push on with chunking?
