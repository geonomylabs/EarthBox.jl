  Log format ‚ÄĒ case4p*.out style logs from @timeit_memit in
  src/utils/PrintFuncs.jl:
  - ++ Finished <section> : cpu(s) : <s> : MB : <RSS_MB> ‚ÄĒ per-section
  - ++ # Finished <group> : cpu(s) : <s> : MB : <RSS_MB> ‚ÄĒ group wrapper

  Key gotcha: Sys.maxrss() is sampled before the section runs, so the printed MB
   is start-of-section RSS, not end. To attribute growth, compute RAM(next) ‚ąí 
  RAM(self) and filter out # lines first (they're group wrappers sampled at a
  different point and create false "drops"). Sys.maxrss() is also
  high-water/monotonic per process ‚ÄĒ real drops only appear when the OS reclaims
   pages.

  Top RAM offenders identified (case4 logs, late April 2026):
  1. looping over nodes to calc nonzeros + rhs ‚ÄĒ Stokes assembly inside Picard
  loop, dominant inner-loop leak
  2. updating topography using lava flow ‚ÄĒ monotonic accumulation, not
  proportional to flow count ‚Üí stateful arrays growing across invocations
  3. managing model output ‚ÄĒ ~19 GB single-step spike from getoutform allocating
   one nmarkers copy per marker scalar field (~27), all held in
  scalars_on_markers until make_jld2_file finishes
  4. calculating Stokes norms ‚ÄĒ MathTools.jl abs.(v_old .- v) allocates
  ~8√ó/Picard iter
  5. updating topography using sediment transport ‚ÄĒ largely addressed by commits
   b2aef8a / 576db23 / 24eb885

  Note that #5 and the recent git history (inside_flags fix in 24eb885, sediment
   transport optimization in b2aef8a) suggest the picture may have shifted since

  What the post-loop simplification buys us for free

  Inside apply_compaction_correction[_for_topography_and_markers], the tail
  computation today is:
  total_uncorrected = sed_init + new_thick_compacted
  total_corrected   = sed_init_compacted + new_thick_decompacted
  topo_corrected    = topo_new - (total_corrected - total_uncorrected)
  return topo_corrected, total_corrected, _

  The caller IMMEDIATELY does copy_array_1d!(topo_gridy_corrected, 
  solver.topo_gridy) and rebinds sediment_thickness_total = total_corrected. So:
  - Write topo_corrected directly into solver.topo_gridy (no buffer needed)
  - Write total_corrected directly into solver.sediment_thickness_total_buffer
  (the buffer that sediment_thickness_total already points to from Phase¬†2)
  - Fold the 4 broadcasts into ONE fused loop:
  for i in 1:toponum
      total_uncorrected_i = sed_init[i] + new_thick_compacted[i]
      total_corrected_i   = sed_init_compacted[i] + new_thick_decompacted[i]
      solver.topo_gridy[i] = topo_new[i] - (total_corrected_i -
  total_uncorrected_i)
      solver.sediment_thickness_total_buffer[i] = total_corrected_i

‚ĚĮ What is the max displacement refering to?                                     

‚óŹ Max displacement comes from this line in
  src/tests/tests/surface_processes/sediment_transport/SedimentTransportTest.jl:

  println(">> Max displacement: ",
          maximum(transport_solver.compaction_displacement_max))

  So it's maximum(solver.compaction_displacement_max) ‚ÄĒ a single scalar in
  meters.

  What compaction_displacement_max represents

  It's a Vector{Float64} of length toponum, computed at the end of the run by
  update_compaction_displacement_max!
  (SedimentTransportSolverManager.jl:368-380):

  sediment_thickness_markers_final = calculate_final_thickness_after_burial(
      porosity_initial, depth_decay_term,
      solver.sediment_thickness_initial,    # original column thickness
      sediment_thickness_total              # total column at end of run
  )
  solver.compaction_displacement_max =
      solver.sediment_thickness_initial - sediment_thickness_markers_final

  So at each x-node of the topography grid:

  compaction_displacement_max[i] = original_sediment_thickness[i] ‚ąí 
  final_compacted_thickness[i]

  It's how much the original sediment column at that x-location was squeezed 
  downward by additional sediment piling on top during the run, using the
  porosity/depth-decay law (Athy compaction). Downstream, this drives
  advect_markers_using_compaction ‚ÄĒ the markers in the original column get
  pushed down by exactly this displacement so they keep tracking the compacted
  geology.

  Why it's a useful regression assertion

  maximum(compaction_displacement_max) is the worst-case compaction at any node
  ‚ÄĒ a single scalar that's a function of every step's evolution of the
  topography (via sediment_thickness_total, which is the running output of every
   inner step's apply_compaction_correction). Any drift in:
  - the diffusion solve (legacy or optimized),
  - the diffusivity grid,
  - the pelagic sedimentation grid,
  - water depth / drainage divides / downstream distances (which feed back into
  next-step diffusivity),
  - the compaction correction itself,

  ‚Ä¶would change sediment_thickness_total, which would change
  sediment_thickness_markers_final, which would change this max. So
  321.3617034366707 (legacy) and 321.40213555158016 (optimized) being
  bit-identical pre- and post-refactor is strong evidence that the entire
  inner-loop chain is preserved end-to-end. It's not just a smoke test ‚ÄĒ it's an
   end-state of a multi-step accumulation.

  Worth noting: it's a single number, so if the per-node trajectories drift but
  happen to produce the same scalar maximum, this check would pass while masking
   a regression. A stronger assertion would compare the full vector
  (copy(transport_solver.compaction_displacement_max)) against a baseline;
  that's what the cross-call regression test would do if we add it.
