‚óŹ The first thing to note is that Sys.maxrss() is a high-water mark, not current
   RSS ‚ÄĒ it only ever increases over the process lifetime. That changes how you
  read this trace.

  What the numbers actually show

  - 55043 MB is flat from start through before scalar loop. That is the
  watermark already set earlier in the run, before make_jld2_file was called.
  The IDs / X / Y writes inside make_jld2_file did not push past it ‚ÄĒ each one's
   transient fit under the existing ceiling, helped by the GC.gc(false) you do
  between them (lines 81, 93, 105).
  - Between before scalar loop (55043) and scalar 5/11 (57102), the watermark
  jumped ~2060 MB. Then it stays pinned at 57102 for 10/11, 11/11, pre-close,
  and post-close ‚ÄĒ not because nothing was freed, but because maxrss cannot go
  down.

  So the spike is a single peak that happened somewhere in scalar iterations 
  1‚Äď5. The probe fires every 5 iterations, so you can't tell which one from this
   trace alone.

  Why the scalar loop spikes when IDs/X/Y didn't

  The IDs, X, and Y blocks each have a GC.gc(false) between them (lines 81, 93,
  105). The scalar loop (lines 109‚Äď118) has no GC between iterations. Each
  iteration:

  1. Allocates a fresh array via getoutform(obj) (line 111),
  2. Hands it to JLD2, which must allocate zlib compression buffers to write it,
  3. Moves on before the previous iteration's array + zlib buffer have
  necessarily been reclaimed.

  Two or three of those overlapping transients stacked is enough to push the
  watermark ~2 GB above the IDs/X/Y pattern ‚ÄĒ which is exactly the gap you see.
  The scalar payloads are likely Float64 per-marker arrays comparable in size to
   X/Y, but X and Y were forced not to overlap; the scalars can.

  Secondary possibility

  One of scalars 1‚Äď5 could be intrinsically larger than Float64-per-marker (e.g.
   a type that getoutform widens, or a composite). The probe granularity hides
  this. If you temporarily probe every iteration instead of every 5th, you'll
  see whether the jump happens in one step (a single fat field) or accumulates
  across several (overlapping transients with no GC). My money is on the latter,
   because the IDs/X/Y evidence shows that a GC.gc(false) between equally-sized
  writes kept the watermark flat.

  What the post-close reading does not tell you

  after jldopen close: 57102.54 MB does not mean JLD2 leaked 2 GB. It only means
   the process touched 57102 MB at some peak moment during the scalar loop.
  Current RSS after close could be much lower; maxrss just can't report that.

‚ĚĮ What could be done to recue this RAM pressure? Do not modify my code yet.     

‚óŹ A few options, roughly ordered by effort-to-payoff:

  Cheap, mirrors what already works

  1. Add GC.gc(false) inside the scalar loop, either every iteration or every
  2‚Äď3. The IDs/X/Y blocks prove this pattern keeps the watermark flat; the
  scalar loop is the only place in this function missing it. Lowest risk,
  smallest change.
  2. Wrap the loop body in a let ... end, matching the IDs/X/Y blocks. A
  let-scoped group and getoutform result become unreachable at the end of the
  block, giving the GC a clearer signal than a bare for body where bindings are
  reused across iterations.

  Points 1 and 2 together are the minimum change that should flatten the 2 GB
  bump.

  Moderate, structurally cleaner

  3. Extract the per-scalar write into a small function ‚ÄĒ write_scalar!(file, 
  obj, meta). When the function returns, every local (the getoutform array, the
  JLD2.Group, any compression temporaries held in caller frames) is unreachable
  by construction. Pair with a GC.gc(false) after the call, or even just rely on
   Julia's GC having a cleaner view of lifetime.
  4. Reusable scratch buffer. If getoutform(obj) always returns the same
  size/dtype (one Float64 per marker, I'd guess), adding an in-place variant
  getoutform!(buf, obj) with a single pre-allocated buf reused across all 11
  scalars eliminates 10 of the 11 allocations outright. This is the cleanest fix
   ‚ÄĒ no GC reliance at all ‚ÄĒ but requires touching getoutform's API. Note JLD2
  will still allocate its own zlib working buffers; the scratch buffer removes
  the per-iteration input allocation, not the compressor's state.

  Tune the compressor

  5. Reduce the JLD2 chunk size for scalar datasets. Zlib's working set scales
  with chunk size; smaller chunks = smaller compressor footprint per write at
  the cost of a slightly larger file and more per-chunk overhead. Worth trying
  only if 1‚Äď4 aren't enough.
  6. Swap Zlib for LZ4 (via CodecLz4). LZ4 has a much smaller internal state
  than zlib and compresses faster; the file gets bigger, and you lose
  Python-earthbox h5py compatibility that your comment on line 64 calls out ‚ÄĒ
  probably a non-starter for you, but mentioning it for completeness.

  The elephant in the room

  7. The baseline is already 55 GB before make_jld2_file is even called. The 2
  GB spike is only interesting because it sits on top of that ceiling. If you're
   chasing overall RAM pressure rather than just this peak, the bigger ROI is
  upstream: marker count √ó enabled fields √ó dtype, and whether
  enabled_marker_objs holds copies or views of marker-table columns. Fixing the
  scalar-loop spike buys you ~2 GB of headroom; trimming the baseline could buy
  tens.

  My recommendation: try (1)+(2) first ‚ÄĒ it's a five-line change that should
  eliminate the 2 GB spike entirely. If it doesn't, (3) is the next step. (4) is
   the "proper" fix if you're optimizing this path seriously.

‚óŹ I've read it. Here are the optimization levers in getoutform, ordered by
  expected RAM impact.

  What allocates today (both 1D and 2D versions are identical structurally):

  data = eb_array.array .* fac1 .+ fac2          # 1 full-size alloc (fused ‚ÄĒ 
  OK)
  ...
  return log10.(abs.(data))                       # 1 more full-size alloc

  - Non-log10 branch: 1 allocation the size of the input.
  - log10 branch: 2 allocations (data, then the log10(abs(¬∑)) result). The
  nested broadcast log10.(abs.(data)) is fused into a single output array, but
  it still reads from data, so data can't be freed until the fused pass finishes
   ‚ÄĒ both exist in RAM at the same time.

  If your typical marker array is Vector{Float64} at N markers, each allocation
  is 8¬∑N bytes. At 100 M markers that's 800 MB per allocation. The 2 GB spike
  you measured is consistent with 2‚Äď3 of these overlapping across scalar
  iterations.

  ---
  Optimization 1 ‚ÄĒ fuse the log10 branch (smallest diff, material win)

  Replace the two-step log10 branch with one fused broadcast so there's no
  intermediate data:

  if use_log10
      return @. log10(abs(eb_array.array * fac1 + fac2))
  else
      return @. eb_array.array * fac1 + fac2
  end

  Cuts the log10 path from 2 allocations to 1. For any scalar field where
  outform.log10 == true, this removes one full-size transient ‚ÄĒ possibly several
   hundred MB per call, depending on which of your 11 scalars use log scaling.

  ---
  Optimization 2 ‚ÄĒ add an in-place variant getoutform!(buf, eb_array) (best RAM
  win)

  Expose a sibling method that writes into a caller-supplied buffer:

  function getoutform!(buf::AbstractVector{Float64},
  eb_array::AbstractEarthBoxArray1D)
      fac1 = eb_array.outform.fac1
      fac2 = eb_array.outform.fac2
      if eb_array.outform.log10
          @. buf = log10(abs(eb_array.array * fac1 + fac2))
      else
          @. buf = eb_array.array * fac1 + fac2
      end
      return buf
  end

  The caller in TimeStep.jl allocates one scratch Vector{Float64} of length
  nmarkers before the loop and reuses it across all 11 scalar writes.
  Allocations drop from 11 to 1. Note JLD2's group["array"] = buf will still
  allocate its own zlib compression buffers internally ‚ÄĒ but the getoutform
  allocation is the biggest per-iteration transient and the one you control.
  Pair with the GC.gc(false) you just added and the scalar-loop watermark should
   flatten close to zero excess.

  Caveat: the 2D variant needs a Matrix{Float64} scratch, or separate methods.
  Also, the caller must stop mutating the returned buffer unless it's safe to
  clobber the scratch ‚ÄĒ your current TimeStep.jl scalar loop doesn't mutate, so
  this is fine there. The X/Y path does ./= 1000.0 and .*= -1/1000 in place on
  the returned array; those sites could keep using the allocating getoutform or
  adopt getoutform! with their own per-call scratch.

  ---
  Optimization 3 ‚ÄĒ narrow the return type

  The annotation is ::Union{Vector{Float64}, Vector{Int64}, Vector{Int16}}, but
  the body always returns Vector{Float64} because fac1::Float64 forces
  promotion. A declared Union defeats inference at call sites (JLD2's
  group["array"] = ... will see a Union and dispatch through a slower path, and
  may box elements in generic paths). Either:

  - Narrow to ::Vector{Float64} (matches actual behavior), or
  - Drop the annotation entirely and let inference see through.

  This is not a RAM win in absolute terms, but it can reduce small per-element
  boxing allocations in downstream code paths that touch the returned array
  generically. Worth doing for correctness regardless.

  ---
  Optimization 4 ‚ÄĒ alias-return for the identity case

  If fac1 == 1.0 && fac2 == 0.0 && !use_log10, you could return eb_array.array
  directly and skip the copy entirely. But this introduces aliasing: if the
  caller ever mutates the return value (the X path does), you'd corrupt the
  source. Fragile ‚ÄĒ I'd skip this unless you audit every call site or add a
  copy_if_needed flag.

  ---
  Recommended order

  1. Apply Opt 1 (fuse log10) ‚ÄĒ it's a 3-line local change, no API impact,
  always a win.
  2. Apply Opt 2 (getoutform! + caller scratch buffer) ‚ÄĒ the structural fix that
   makes the scalar loop's RAM cost independent of field count.
  3. Apply Opt 3 (return type) alongside ‚ÄĒ free cleanup.

  (1) + the GC.gc(false) you already added likely closes most of the 2 GB gap.
  (2) is what you reach for if it doesn't, or if you want a principled rather
  than GC-dependent solution.
