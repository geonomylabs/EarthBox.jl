  ---
  RAM Optimization Plan: Streaming Marker Output

  Root Cause (confirmed from prior analysis)

  The 19 GB spike is caused by GetMarkerData.jl:30–44, which calls
  getoutform(obj) for all 25 enabled scalar fields in a loop and accumulates all
   the resulting arrays simultaneously in scalars_on_markers — keeping them
  alive until make_jld2_file finishes writing. Peak RAM scales as N_fields × 
  N_markers × 8 bytes. At ~60M markers and 25 fields that is ~12 GB of
  duplicated data, plus coordinate arrays.

  The JLD2 backup is not the culprit (it writes direct array references, no
  copies).

  ---
  The Fix: Stream One Field at a Time

  Current pipeline:
  1. Loop over 25 fields → call getoutform(obj) → store each result in
  ScalarField struct → push all into scalars_on_markers
  2. make_jld2_file writes all 25 arrays (all still alive in memory)
  3. Peak = base + 25 × N_markers copies

  New pipeline:
  1. Loop over 25 fields → store only metadata (name, units, number_type) + a
  reference to the original array object in two lightweight lists
  2. make_jld2_file loops over the array object references: compute getoutform →
   write to JLD2 → the result goes out of scope → one copy at a time
  3. Peak = base + 1 × N_markers copies (the current field being written)

  XDMF string generation only needs field names/units/types — not the data 
  arrays — so the XML output is unaffected.

  ---

  struct ScalarFieldMeta

  Before: 47,356 MB → 66,714 MB = +19,358 MB spike
  After: 47,357 MB → 55,318 MB = +7,961 MB spike

  That's an 11.4 GB reduction (59% improvement). The output now takes roughly
  the same wall time (~170 s vs ~167 s), so no regression there.

  ---
  Why ~8 GB still remains

  The remaining spike has a few sources:

  1. marker_xy_km + marker_ids live in Markers2djld for the whole write — these
  arrays are pre-computed and stored in the struct before make_jld2_file opens
  the file. At ~60M markers: 960 MB + 480 MB = ~1.44 GB permanently held
  throughout the scalar write loop.
  2. marker_x_m + marker_y_m from make_marker_arrays_for_xdmf_format — both are
  alive simultaneously while marker_xy_km is being filled, then go out of scope.
   GC may not collect them before the scalar loop starts: ~960 MB.
  3. ZlibCompressor buffers — at level 1, the compressor needs working memory
  roughly equal to the input array size per write. Each 60M-element scalar is
  ~480 MB, so peak during each write includes the source + compression buffer:
  ~480–960 MB per field.
  4. GC non-determinism — Julia's GC doesn't guarantee immediate collection when
   things go out of scope, so some freed arrays may still count toward maxrss at
   the moment the peak is measured.

  ---
  Further reduction options (if needed)

  The next meaningful step would be to eliminate the pre-computed
  marker_xy_km/marker_ids from the struct — instead compute and write them
  directly inside the JLD2 block before the scalar loop, then let them go out of
   scope. This would save ~1.44 GB of persistent allocation during the scalar
  writes. It's a more invasive refactor (breaking apart Markers2djld or passing
  raw marker objects into make_jld2_file). Let me know if you'd like to pursue
