  General Julia causes:

  - Heap fragmentation: Julia's GC collects objects but doesn't always return
  pages to the OS. RSS can appear to double even when live data is stable. Run
  GC.gc() periodically or profile with --track-allocation to confirm.
  - Type instability: model.melting.parameters...value field accesses may be
  type-unstable if value is typed Any, causing boxing on every access — check
  with @code_warntype.
  - Growing Vectors: any push! without a pre-sized buffer doubles capacity on
  resize.

  check what PartialMeltIndices.get_partial_melt_marker_indices returns — if
  it's a dynamically-sized Vector grown with push!, that's likely the dominant
  allocator.

● Now I have a clear, verified picture. Here are the definitive findings ranked
  by impact:

  ---
  #1 — BackupTransportArrays.jl lines 48–64 and 159–163 (PRIMARY CAUSE)

  Every timestep, copy() allocates 20 brand-new large grid arrays. The old
  *0.array pointer is just overwritten, so the GC must collect it. Before it
  does, you hold 2× the memory for all those arrays simultaneously. Over
  thousands of timesteps, the heap expands and GC fragmentation prevents it from
   contracting.

  Fix — change all copy() to copyto!() into pre-allocated backup arrays. The *0
  arrays already exist in model, so the assignment isn't needed either:

  # Before (allocates new array every timestep):
  model.stokes_continuity.arrays.viscosity.etas0.array =
  copy(model.stokes_continuity.arrays.viscosity.etas1.array)

  # After (writes into existing array, zero allocation):
  copyto!(model.stokes_continuity.arrays.viscosity.etas0.array,
  model.stokes_continuity.arrays.viscosity.etas1.array)

  This one change likely explains most of the observed doubling.

  ---
  #2 — MagmaBody.jl lines 269–293, calculate_marker_indices_mantle_search_domain

  Two vectors allocated per timestep per drainage basin:
  - marker_indices_tmp = Vector{Int64}(undef, marknum) — full marker array size
  - marker_indices_mantle_injection_search_domain = Vector{Int64}(undef, 
  nmarkers_injection_domain) — trimmed copy

  Fix — preallocate marker_indices_tmp once in model and pass it as a buffer.
  The trimmed output can write into a preallocated array with a count returned
  separately, eliminating the second allocation.

  #3 — PartialMeltIndices.jl line 25, get_partial_melt_marker_indices
